C 和 C++ 的基本区别

C++ 是 C 的扩展，增加了许多新特性。

C 是面向过程的语言，适合底层开发；C++ 是 C 的超集，支持面向对象编程，功能更强大。

面向过程开发

首先，什么是面向过程：

把程序看成一系列步骤，一步步去执行。

就像做菜，按照食谱一步一步来：洗菜 → 切菜 → 炒菜 → 装盘。

它的程序由函数组成的，其中数据（变量）和操作（函数）是分开的，适合简单、线性的任务。

面向对象开发

C++ 的风格是面向对象编程，把程序看作一组对象，对象有自己的属性和行为。就像开餐厅，餐厅里有厨师、服务员、顾客等对象。每个对象都有自己的职责：

  • 厨师：负责做菜。
  • 服务员：负责上菜。
  • 顾客：负责点菜和吃饭。

其实有点像 python 中一个个类

它的程序由类和对象组成，数据（属性）和操作（方法）封装在一起，适合复杂的系统，易于扩展和维护。

C++ 的面向对象特性

C++ 支持面向对象编程，主要有以下特性：

类（Class）

  • 类是对象的模板，定义了对象的属性和行为。
  • 比如，Dog 类可以有属性 name 和方法 bark()。

封装（Encapsulation）

  • 把数据和方法封装在一起，隐藏内部细节。
  • 比如，你不需要知道手机内部如何工作，只需要知道怎么用。

继承（Inheritance）

  • 一个类可以继承另一个类的特性。
  • 比如，Cat 和 Dog 都可以继承 Animal 类的特性。

多态（Polymorphism）

  • 同一个方法在不同对象中有不同的行为。
  • 比如，Animal 类的 speak() 方法，Dog 会“汪汪”，Cat 会“喵喵”。

总结

  • C：面向过程，适合底层开发，代码简单直接。
  • C++：支持面向对象，适合大型项目，代码更易扩展和维护。
  • 面向过程：关注步骤和流程。
  • 面向对象：关注对象和它们之间的关系。

unmae -m

本命令用于查询CPU的系统架构，一般有x86,x86_64,amd64,arm64,arrch64

架构 指令集位数 原因
X86 32位 英特尔出的处理习惯以86为结尾，如80186、80286、80386，所以之后被称之为 X86
AMD64 64位 32位向64位演进时，AMD 抢先制造出了兼容 X86 32位指令集的 64位处理器
X86-64 64位 英特尔设计 IA-64，比amd晚了一步，且不兼容 X86 32位指令集，比较惨淡
因此后续因特尔也采用 AMD64，但在此基础上进行了扩充，并改名为 X86-64
所以 x86_64,x64,AMD64基本上是同一个东西
现在用的intel/amd的桌面级CPU基本上都是x86_64
ARM 一般用于嵌入式系统
ARM-V8 AArch64：64位执行状态
AArch32：32位执行状态		 有两种执行态，适用于 32位和64位
AArch64 64位 从 ARM-V8 中独立出来

可以简单的将 x86_64,x64,AMD64理解为同一种64位的CPU架构，arrch一般用于嵌入式系统。

git 命令详解

发表于 2025-02-05

前言

主要整理一下我日常使用的时候，常用的命令

git init

在本地创建一个新项目时，将其初始化为 Git 仓库。

git remote add origin <远程仓库地址>

将本地仓库与远程仓库关联，从而将本地仓库推送到 GitHub 或 GitLab 等远程仓库时使用。

下面是对这个命令进行详细的解释：

  • origin：这是远程仓库的名称。origin 是一个默认的名称，通常用于指向主远程仓库。你可以使用其他名称，但 origin 是最常用的。
  • 仓库地址可以是 HTTPS 格式和 SSH 格式，这里强烈建议使用 ssh 链接，不要使用 HTTPS，反正我是经常报错，连接不上。但是使用 SSH 连接的情况下，一般都很稳定。

git remote -v

查看当前仓库关联的远程仓库地址

示例

这是我在我对终端里面的实际输出，不难发现，我使用的是使用 SSH 格式。

  • fetch：这表示该 URL 用于从远程仓库拉取（fetch）数据。当你运行 git fetch 或 git pull 时，Git 会使用这个 URL 从远程仓库获取最新的更改。
  • push：这表示该 URL 用于向远程仓库推送（push）数据。当你运行 git push 时，Git 会使用这个 URL 将你的本地更改推送到远程仓库。

git branch 分支管理

用于查看本地分支列表，，Git 会列出当前仓库中的所有本地分支，并在当前分支前加上一个星号（*）

我的仓库中存在两个分支，main 和 master ，而且当前分支是 main。

接下来我们顺便拓展一下其他相关的命令

创建一个新分支

git checkout <分支名>

切换到指定分支

创建并切换到新分支，其实就是对前面两个命令的总结

至于删除远程分支，这个我觉得谨慎点，我还是在 github 上操作吧

提交与推送

这个经常用到

git add <文件>

将文件添加到暂存区

其中 . 表示添加所有文件，不过我一般喜欢使用 git status 来判断哪些文件发生了更改，然后仅仅提交更改后的文件，也是可行之道。

git commit -m "<提交信息>"

将暂存区的文件提交到本地仓库

注意, 后面的引号千万不要搞成中文, 否则按下 enter 键后，会出现小尖尖。

git push -u origin <分支名>

将本地分支推送到远程仓库

这个命令我们需要详细解释一下，首先它是把本地的分支推送到远程 main 分支。

-u 选项是将远程分支 origin/main 设置为本地 main 分支的上游分支，这意味着在后续的 git pull 和 git push 操作中，Git 会自动使用 origin/main 作为目标分支，而不需要每次都指定远程分支名称。

也就是后面我们可以直接使用 git push 和 git pull

这里我们要区分一下 git push origin main ，它也是将本地的 main 分支推送到远程仓库的 main 分支，但是它 不设置上游分支：这个命令不会设置上游分支。每次推送时，我们需要明确指定远程分支名称

git push --force

强制推送，这个是，怎么说呢，非常棒的命令。

首先我们介绍一下什么时候使用这个命令，一般情况下，远程仓库是本地仓库的子集，也就是，远程仓库有的，本地仓库肯定都有，当然这个子集是广泛的概念。但是，当你想要大幅度改变远程仓库的内容时，如果只是使用 git push ，包报错的，老弟，报错输出我没了，当时我是想把整个仓库的内容都换了，相当于删库，但是我懒得创建新库了，推送的时候就遇到了这个问题。只能说 github 也太贴心了。

当然，你要在远程仓库里设置，允许强制推送。
